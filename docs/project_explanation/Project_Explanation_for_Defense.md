# Объяснение проекта DiplomaWork

## 1. Что это за проект

DiplomaWork — это система быстрого поиска лиц в биометрической базе данных. Она состоит из backend-сервиса на FastAPI, desktop-клиента на PySide6, подсистемы векторного поиска на FAISS, слоя хранения метаданных и эмбеддингов, а также исследовательской ветки для обучения и оценки собственных моделей.

Практическая цель системы — не просто распознать лицо на изображении, а сделать это быстро, воспроизводимо и в форме операторского инструмента. Пользователь может загрузить фотографию, запустить поиск, сравнить результаты между pipeline, просмотреть записи в базе, проверить состояние индекса и использовать live webcam режим.

Ключевая идея проекта в том, что поиск по лицам разбивается на две независимые части:

- извлечение биометрического признака, то есть embedding лица;
- быстрый поиск ближайших embedding по векторному индексу.

За счёт этого backend не завязан жёстко на одну модель и поддерживает несколько pipeline.

## 2. Общая архитектура

Проект разделён на несколько крупных модулей:

- `backend/` — API, бизнес-логика, доступ к БД, управление индексом, runtime pipeline;
- `desktop/` — операторский клиент на PySide6;
- `training/` — исследовательская ветка для обучения, оценки и подготовки датасетов;
- `deploy/model_bundle/` — внешний bundle с готовым индексом и связанными артефактами;
- `scripts/` — служебные скрипты импорта, расчёта эмбеддингов, benchmark и обслуживания.

Архитектурно система устроена так:

1. Desktop отправляет изображение или кадр с вебкамеры в backend.
2. Backend валидирует входной файл.
3. Pipeline извлекает одно или несколько лиц и считает embedding.
4. FAISS индекс ищет ближайшие векторы.
5. Backend достаёт metadata из БД и формирует ответ.
6. Desktop показывает summary, список совпадений, detected faces и технические детали.

Такой подход удобен потому, что UI, API и индексный слой не обязаны знать детали конкретной модели.

## 3. Backend

Backend построен на FastAPI и инициализируется в `backend/app/main.py`. На старте создаётся `PipelineRegistry`, который собирает доступные runtime pipeline, поднимает extractors, index manager для каждого pipeline и пытается загрузить последние snapshots индекса.

Основные маршруты API:

- `GET /v1/health` — состояние backend и pipeline;
- `POST /v1/enroll` — регистрация человека;
- `POST /v1/search` — поиск по одному выбранному pipeline;
- `POST /v1/search/compare` — параллельное сравнение двух pipeline;
- `GET /v1/persons` — просмотр записей;
- `GET /v1/index/stats` — статистика индекса;
- `POST /v1/index/rebuild` — пересборка индекса.

Backend отдаёт не только matches, но и информацию о найденных лицах в кадре. Это особенно важно для multi-face search и live webcam режима.

## 4. Runtime pipeline и модели

В проекте используется pipeline abstraction. Это означает, что логика поиска не зависит напрямую от одной конкретной модели.

Сейчас в системе предусмотрены два логических pipeline:

- `pretrained` — baseline pipeline;
- `custom` — сравнительный или исследовательский pipeline.

В health-ответе backend также сообщает:

- какой pipeline является default;
- какие pipeline доступны;
- какой backend используется;
- имя модели;
- включён ли strict single-face enroll;
- включён ли multi-face search.

Смысл такого разделения:

- baseline нужен как стабильный эталон;
- custom нужен для исследовательской части и сравнительных запусков;
- compare режим позволяет сравнивать задержку и поведение pipeline на одном и том же входе.

На практике в проекте уже есть несколько модельных направлений:

- ONNX baseline на базе `buffalo_l`;
- custom comparative branch;
- training-ветка для собственных экспериментов;
- внешний `deploy/model_bundle` как отдельный bundle артефактов.

Это важно правильно формулировать: UI и API поддерживают несколько pipeline, а не одну жёстко зашитую модель.

## 5. Поиск по изображению

Поиск — это главный сценарий проекта. Он реализован в backend через `backend/app/api/routes/search.py`.

Порядок работы поиска:

1. backend получает изображение;
2. extractor находит лица на изображении;
3. для каждого лица вычисляется embedding;
4. embedding отправляется в FAISS индекс;
5. индекс возвращает top-k ближайших совпадений;
6. backend дополняет их person metadata из БД;
7. формируется ответ с score, distance, pipeline, label, face index и detected faces.

В системе есть threshold-based decision logic:

- если лучший score выше порога, решение интерпретируется как `match`;
- если ниже — как `unknown`.

Это важно, потому что ranking и финальное решение не одно и то же. Система может показать ближайших кандидатов, но при этом честно сообщить, что уверенность недостаточна.

## 6. Регистрация человека (Enroll)

Регистрация человека отличается от поиска более строгими правилами. Здесь важно не загрязнить базу.

Для enroll действует strict single-face policy:

- если на изображении 0 лиц, backend возвращает `422`;
- если на изображении больше 1 лица, backend тоже возвращает `422`;
- если файл вообще невалиден, возвращается `400`.

То есть enroll предназначен для контролируемой регистрации одного человека, а не для сцен с несколькими лицами.

При успешной регистрации система:

1. создаёт или использует запись `person`;
2. сохраняет embedding;
3. обновляет индекс соответствующего pipeline.

Смысл такого ограничения простой: каждая зарегистрированная запись должна соответствовать одному чистому биометрическому образцу.

## 7. Multi-face search

В отличие от enroll, поиск допускает несколько лиц на одном изображении.

Это означает:

- backend может найти несколько лиц;
- по каждому лицу может быть выполнен отдельный поиск;
- desktop показывает detected faces и связывает matches с конкретным лицом.

Поэтому в интерфейсе присутствует отдельная логика per-face результата. Это делает систему более пригодной для реальных сцен, где на фото или в кадре вебкамеры может быть несколько человек.

Главное правило, которое удобно проговорить на защите:

- enroll — строгий и чистый;
- search — аналитический и допускает multiple faces.

## 8. Почему используется FAISS, а не только БД

В проекте БД и индекс выполняют разные задачи.

База данных хранит:

- записи о людях;
- embeddings;
- snapshots индекса;
- service metadata.

FAISS отвечает за быстрый nearest-neighbor поиск по 512-мерным векторам.

Причина, почему поиск не выполняется напрямую через обычную SQL БД:

- хранить embeddings в БД удобно;
- но искать ближайшие векторы через обычный SQL на больших объёмах неэффективно.

Поэтому принята нормальная инженерная схема:

- metadata и хранение — в БД;
- поиск похожих embedding — в FAISS.

Это один из центральных инженерных тезисов проекта.

## 9. Что именно хранится в системе

По умолчанию сервер не хранит исходные изображения лиц как основное operational storage. Это важный проектный принцип.

В системе хранятся:

- `person`;
- `embedding`;
- `index snapshot`.

Такой подход уменьшает объём хранения, упрощает privacy narrative и делает storage слой более специализированным под задачу поиска.

Если объяснять очень просто:

- фото нужны для извлечения embedding;
- дальше для поиска нужен уже embedding, а не сам JPEG.

## 10. Индекс и его обслуживание

Система умеет не только искать, но и обслуживать индекс.

Основные возможности:

- просмотр статистики индекса;
- rebuild индекса;
- работа с несколькими pipeline;
- загрузка snapshot на старте backend.

Это нужно для практической эксплуатации. В дипломном проекте часто показывают только поиск, но здесь есть ещё и operator-level maintenance.

В проекте используются разные варианты индекса, включая:

- HNSW;
- IVF-PQ;
- другие параметры approximate nearest-neighbor конфигурации.

Такой дизайн позволяет сравнивать точность и скорость, а не быть привязанным к одному фиксированному индексу.

## 11. Desktop-клиент

Desktop-клиент реализован на PySide6 и является операторским интерфейсом системы. Он не просто вызывает backend, а организует полный пользовательский workflow.

Текущая структура страниц:

- `Dashboard`;
- `Face Search`;
- `Database`;
- `Logs`.

Это важно, потому что старое описание в терминах отдельных табов `Search / Enroll / Persons / Stats` уже не полностью соответствует актуальному интерфейсу. Сейчас интерфейс стал более цельным.

### 11.1 Dashboard

Dashboard — это не декоративная панель, а краткая сводка по системе. На нём показываются:

- backend connectivity;
- pipeline availability;
- summary metrics;
- recent activity;
- system overview.

Это входная точка для оператора перед запуском поиска или регистрации.

### 11.2 Face Search

Face Search — главный экран приложения. Он организован как workflow:

1. выбрать изображение или запустить камеру;
2. выбрать режим;
3. запустить действие;
4. посмотреть summary;
5. изучить top matches и detected faces.

На этом экране есть несколько режимов:

- `Search`;
- `Enroll`;
- `Compare`.

Также здесь есть:

- image preview / drop zone;
- список найденных кандидатов;
- detected faces;
- advanced options;
- technical details.

Secondary controls специально вынесены в менее заметные секции, чтобы основной пользовательский путь не был перегружен.

### 11.3 Database

Database page показывает список записей и детали выбранного person. Это не просто dump таблицы, а операторский экран просмотра и обслуживания базы.

Здесь можно:

- просматривать записи;
- видеть metadata;
- удалять записи;
- работать со списком как с operational registry.

### 11.4 Logs

Logs page нужен для сопровождения и отладки.

На нём оператор видит:

- backend state;
- pipeline state;
- recent events;
- index tools;
- maintenance actions.

То есть это экран сопровождения системы, а не только отладочный экран разработчика.

## 12. Live webcam режим

Одна из сильных практических функций проекта — live webcam search.

Нужно формулировать это точно: это не тяжёлый постоянный видеострим на backend, а near real-time operator-assisted workflow.

Как это работает:

1. desktop показывает локальный preview камеры;
2. периодически выбираются кадры;
3. кадры отправляются в backend;
4. backend выполняет detection, embedding extraction и search;
5. desktop визуализирует найденные лица и результаты поиска.

Преимущества такого подхода:

- меньше нагрузка, чем при непрерывной передаче потока;
- проще локальный запуск;
- удобнее для MVP и демонстрации;
- понятный сценарий для оператора.

Live webcam особенно хорошо демонстрирует:

- multi-face search;
- compare mode;
- latency differences;
- визуальный overlay найденных лиц.

## 13. Compare mode

Compare mode нужен для параллельного сравнения двух pipeline на одном и том же входном изображении.

Это полезно сразу по нескольким причинам:

- можно сравнивать задержку;
- можно сравнивать score и decision;
- можно видеть различие в поведении моделей;
- можно использовать baseline как эталон, а custom как исследовательскую ветку.

Именно compare mode делает проект более инженерным: система не только выдаёт ответ, но и позволяет анализировать качество pipeline.

## 14. База данных

Проект поддерживает два режима хранения:

- PostgreSQL, обычно через Docker Compose;
- SQLite, обычно для локальной разработки и быстрых запусков.

Это даёт удобный компромисс:

- PostgreSQL — для нормального рабочего окружения;
- SQLite — для быстрой локальной отладки и автономного демо.

Важно понимать, что смена БД не меняет общую архитектуру: API, desktop и FAISS логика остаются прежними.

## 15. Скрипты и служебная автоматизация

В проекте есть набор скриптов, которые облегчают обслуживание и эксперименты.

Основные сценарии:

- вычисление embedding по датасету;
- benchmark поиска;
- rebuild индекса;
- импорт внешнего bundle;
- запуск через PowerShell helper;
- обучение и оценка собственной модели.

Практический смысл этих скриптов:

- они делают проект воспроизводимым;
- позволяют быстро повторять эксперименты;
- сокращают количество ручных шагов;
- помогают не терять исследовательскую часть за пределами UI.

## 16. Datasets и training-ветка

Помимо runtime части, в репозитории есть исследовательская training-ветка.

Она нужна для:

- подготовки датасетов;
- обучения своей модели;
- smoke-run и evaluation;
- исследовательских экспериментов.

В проекте фигурировали и использовались, в зависимости от сценария, такие наборы и форматы:

- identity-based folders в формате `dataset/<identity>/*.jpg`;
- CelebA;
- DigiFace-1M и локально подготовленные подмножества;
- внешние подготовленные bundle и таблицы соответствия.

Это надо объяснять аккуратно:

- production runtime и training subsystem — не одно и то же;
- training-ветка нужна для исследований;
- runtime-часть нужна для воспроизводимого рабочего поиска.

## 17. deploy/model_bundle

В проекте присутствует `deploy/model_bundle`, который содержит готовые артефакты, связанные с внешней custom веткой.

Там есть:

- `train.csv`;
- `faiss.index`;
- `meta.json`;
- `best.pt`.

Это важно не путать с базовым runtime path. По сути, это отдельный bundle артефактов, который можно использовать для comparative или import сценариев.

Также в проекте есть `scripts/import_bundle.py`, который позволяет импортировать предварительно подготовленные embeddings из bundle в backend DB и использовать их как предзагруженную галерею.

Для защиты безопасная и точная формулировка такая:

- baseline runtime уже стабильно работает;
- custom bundle существует как отдельная comparative/research ветка;
- архитектура проекта допускает такие разные источники embedding и индексов.

## 18. Startup и режимы запуска

Проект можно запускать несколькими способами.

### 18.1 Локально

Локальный сценарий нужен для разработки, быстрой отладки и автономной демонстрации.

Обычно поднимаются:

- backend через `uvicorn`;
- desktop локально;
- SQLite или локальная PostgreSQL конфигурация.

### 18.2 Через Docker Compose

Docker Compose используется для более воспроизводимого окружения. Через него удобно поднимать:

- backend;
- db;
- mounted model assets;
- mounted training outputs.

Также в проекте есть PowerShell helper `tools/run_all.ps1`, который упрощает запуск в Windows-first сценарии.

## 19. Сильные стороны проекта

С инженерной точки зрения сильные стороны проекта такие:

- разделение runtime и training части;
- независимость UI от конкретной модели;
- FAISS как быстрый векторный слой;
- operator desktop, а не только REST API;
- поддержка compare mode;
- live webcam сценарий;
- strict single-face enroll;
- multi-face search;
- индекс как отдельный управляемый компонент;
- несколько режимов запуска.

То есть проект не ограничивается “одним скриптом распознавания лица”, а выглядит как полноценная система.

## 20. Ограничения и честные слабые места

На защите лучше проговаривать ограничения спокойно и технически честно.

Текущие ограничения:

- quality разных custom веток ещё требует дополнительной валидации;
- live webcam — это near real-time, а не тяжёлый промышленный video stream;
- training и deployment custom модели пока не являются полностью завершённой production цепочкой;
- часть сравнительных веток находится в исследовательском состоянии.

Такая формулировка нормальна. Для MVP и pre-defense это не проблема, если ты чётко показываешь, что рабочий baseline и сама архитектура уже функционируют.

## 21. Что говорить на предзащите одной фразой

Если нужно короткое, но правильное описание, его можно сформулировать так:

“Наш проект — это система быстрого поиска лиц в биометрической базе данных. Она состоит из FastAPI backend, PySide6 desktop-клиента, векторного индекса FAISS и слоя хранения metadata и embeddings. Система поддерживает строгую регистрацию одного лица, multi-face поиск, live webcam режим, несколько pipeline и compare mode для сравнительного анализа моделей.”

## 22. Что показывать на демонстрации

Оптимальный порядок демонстрации:

1. открыть Dashboard и показать, что backend online;
2. открыть Face Search и выполнить обычный search;
3. показать enroll на одном лице;
4. показать compare mode;
5. показать Database;
6. показать Logs и index tools;
7. при стабильном состоянии показать webcam.

Это создаёт понятную логику показа: от общей готовности системы к реальному рабочему сценарию.

## 23. Итог

DiplomaWork — это не набор разрозненных файлов, а система с понятной архитектурой и ролями компонентов. Backend отвечает за биометрическую обработку и поиск, desktop обеспечивает операторский workflow, FAISS даёт быстрый поиск по embedding, а база хранит metadata и сами embedding. Дополнительные pipeline, research scripts, training subsystem и bundle tooling делают проект пригодным не только для демонстрации, но и для дальнейшего развития.
